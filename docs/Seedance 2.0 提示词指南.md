# 通用 Seedance 2.0 提示词指南
### 将任何场景转化为逐镜头电影级提示词的系统

本指南旨在教您如何编写最高水平的 Seedance 2.0 视频提示词。它适用于任何人、任何场景以及任何时长。将本指南与您的场景描述一起提供给 Claude（或任何功能强大的大语言模型），您将获得符合以下标准的高质量提示词。

---

## 如何使用本指南

当用户描述他们想要生成的场景时，请按顺序执行以下步骤：

1. **理解场景。** 谁在场景中、在什么地方、发生了什么、情感基调是什么、时长应该是多少、包含多少个镜头。
2. **非必要不提问。** 只有在缺失关键要素（时长、角色数量、基调）时才进行澄清提问。不要询问可以通过合理推断得出的内容——直接做出选择并予以说明。
3. **选择镜头数量。** 将节奏与情感意图相匹配（见下表）。
4. **填写主模板。** 每次都必须按顺序填写每一个板块。
5. **编写镜头。** 每个镜头都要包含画面构图、镜头焦段、相机运动和散文式的动作描述。
6. **在交付前应用检查清单。**

您的唯一任务是将用户的设想转化为精准、AI 可读的电影级提示词。不要强加您自己的故事创意。如果他们提供的内容奇特或稀少，请顺应他们的思路，而不是去无端猜测。

---

## 黄金法则：尊重时间限制

**短时长视频的容量非常有限。放轻松。**

- 5 秒 = 一个简单的节拍
- 10 秒 = 两到三个节拍
- 15 秒 = 最多三到四个节拍

如果一个镜头描述了 5 个动作，画面就会出现崩坏（glitch）。如果在 2 秒内塞进 3 句台词，就会显得非常局促。计算动作数量，计算字数，务必切合实际。

当用户说 **“太多了” (too much)**、**“放轻松” (chill)** 或 **“切合实际一点” (be realistic)** 时——这就是动作或对话过于拥挤的信号。直接删减，不要重写。

## 主模板

仅限纯文本。最终输出中不得包含 Markdown 格式。

```
FORMAT: [时长]s / [镜头数量] SHOTS / [单行概念]

SUBJECT: @image_1. [年龄、体型、发型、特征、活力/性格]

SECONDARY SUBJECT: @image_2. [如果适用，提供相同水平的细节描述]

WARDROBE @image_1: [具体单品、颜色、配饰]
WARDROBE @image_2: [如果适用，提供相同描述]

HERO PROPS: [命名的物品。如果引用了参考图，请标记为 @image_N。]

ENVIRONMENT: [地点、一天中的时间、感官细节。如果存在多个环境，请标记为 (A)、(B)、(C)。]

MOOD: [情感起伏弧线，而不仅仅是某种氛围]

MUSIC: [配乐如何演变——如果附带了预先生成的音频，则标记为 @audio_N]

COLOR LOGIC: [主导色调 + 一种点缀色/流行色]

STYLE: [美学参考 + 技术规格——景深、噪点、光照]

LOGIC RULE: [连贯性规则、防重复规则、道具一致性]

NEGATIVE PROMPT: [可选——需要避免的具体事物]

---

SHOT 1 — 0:00 to 0:0X, [画面构图], [镜头焦段]mm, [相机运动].
[散文式的动作描述。台词以内联方式写入。]

SHOT 2 — 0:0X to 0:0Y, [画面构图], [镜头焦段]mm, [相机运动].
[动作描述]
```

## 严格的格式规则

| 格式规则 | 正确示例 | 错误示例 |
|---|---|---|
| 图像标记 (Image tags) | `@image_1` | `<<<image_1>>>`, `[image_1]`, `**image_1**` |
| 输出中的粗体 (Bold in output) | Plain text (纯文本) | `**bold text**` (Markdown 粗体) |
| 镜头元数据分隔符 (Shot metadata separator) | 句号 + 换行 (Period + line break) | 元数据与动作之间使用 `/` |
| 时间戳 (Timestamps) | 整数秒 (`0:00 to 0:02`) | 小数秒 (`0:01.5 to 0:03.2`) |
| 板块分隔符 (Section dividers) | 元数据与镜头之间使用 `---` | 无分隔符 / 密密麻麻的文本墙 |
| 空行 (Blank lines) | 镜头之间留一个空行 | 所有内容堆叠在一起 |
| 角色标记 (Character tags) | 每个角色都有自己专属的 `@image_N` | 将次要角色也重复标记为 `@image_1` |

**标记规则：** `@image_N` 标记严格用于用户上传的参考图像。一张参考图 = 一个标记。如果角色没有参考图，则不使用标记——请改用散文式文字对其进行描述。

## 时长与镜头数量计算

每个镜头都需要呼吸的空间。模型每秒无法连贯地处理超过约 2-3 个不同的动作。

| 镜头数 | 最佳平均单镜头时长 | 场景类型 |
|---|---|---|
| 1 (长镜头/独角戏) | 完整场景 | 单一连续表演、Vlog 第一视角 (POV)、情绪崩溃、一镜到底的打斗、歌舞环节 |
| 2–3 | 每个 3 秒以上 | 缓慢的氛围感、情绪化的揭示、沉思 |
| 4–6 | 每个约 2-3 秒 | 标准叙事——铺垫、转折、结局 |
| 7–9 | 每个约 1.5-2 秒 | 对话驱动、电影感故事 |
| 10–14 | 每个约 1-1.5 秒 | 快速蒙太奇、MTV 剪辑风格、Vlog 节奏 |
| 15+ | 每个小于 1 秒 | 极不推荐——模型在低于 1 秒时表现挣扎 |

**公式：** `平均单镜头时长 = 总时长 ÷ 镜头数`

如果计算得出的单镜头时长低于 1 秒，请删减镜头或延长总时长。

### 根据情感意图选择镜头数量

- **持久的情感表达**（悲伤、狂喜、紧张）→ 镜头更少，每个镜头时间更长
- **能量与动量**（动作、派对、蒙太奇）→ 镜头更多，节奏更快
- **单一连续表演**（唱歌、打斗、说话、Vlog）→ 1 个长镜头（Oner）
- **带有节拍的叙事**（铺垫 → 冲突 → 高潮）→ 将镜头数量与叙事节拍相匹配

## 音频标记——当用户上传预生成的音频时

这是至关重要的，它会改变镜头的编写方式。

### 如果用户附带了音频文件（歌曲、旁白、对话轨道）：

1. 在 `MUSIC` 板块中将其标记为 `@audio_1`（如果它是人声，也可以在镜头描述中内联标记）。
2. **切勿将完整的音频内容转录到镜头描述中。** 模型会自动将嘴唇动作与音频本身进行同步。
3. 保持镜头描述极简——描述**表演**，而不是歌词/字词。

**示例——附带歌曲的唱歌场景：**
```
MUSIC: @audio_1
SHOT 1 — 0:00 to 0:15, MCU to MS, 50mm, slow pull-back.
画面从紧凑的特写开始，展现 @image_1 在舞台上激情澎湃地表演 @audio_1。摄像机缓慢向后拉，展现出台下的观众。
```

不要写“他唱着‘We will rock you’”或描述歌词节拍。音频本身会驱动表演。

### 混合情况——音频中仅包含用户的对话，但其他角色需要做出回应：

当附带的音频仅包含 @image_1 的声音，但场景需要另一个角色进行回话时，**仅**转录做出回应的角色的台词。将用户的台词标记为 `@audio_1`。

**示例：**
```
@image_1 说话 @audio_1 "Excuse me, do you guys have Rolexes?"
@image_2 回答: "Are you wearing those recording glasses?"
@image_1 回答 @audio_1 "Yeah, I'm recording."
```

### 第一视角 (POV) 场景的音频真实感

当场景是通过真实世界设备（手机、智能眼镜、家用摄像机、头盔摄像头）拍摄的 POV 时，请添加此注释：*“所有音频听起来就像是由设备的麦克风捕获的——自然、略显低沉、没有经过录音室后期处理。”* 否则，生成的音频会感觉低音过重，不符合该格式的真实感。

## 镜头语言

### 画面构图

| 术语 | 含义 | 最适用于 |
|---|---|---|
| ECU | 极特写 (Extreme Close-Up) | 眼睛细节、物体纹理 |
| CU | 特写 (Close-Up) | 面部和颈部 |
| MCU | 中特写 (Medium Close-Up) | 头部和肩膀——对话的主力构图 |
| MS | 中景 (Medium Shot) | 腰部以上——动作的主力构图 |
| WS | 全景/远景 (Wide Shot) | 环境中的全身展现 |
| OTS | 过肩镜头 (Over-The-Shoulder) | 对话、揭示 |
| POV | 第一视角 (Point of View) | 沉浸式主观视角——相机即眼睛 |

### 镜头焦段

| 焦段 | 感觉 | 最适用于 |
|---|---|---|
| 24–28mm | 宽广、沉浸感 | 动作、环境交代、动态空间 |
| 35mm | 纪实、自然 | 手持真实感、街头场景 |
| 50mm | 中性/标准 | 最通用 |
| 85mm | 亲密、浅景深 | 面部、情感表达、人像 |
| 100mm macro | 极致细节 | 微距物体、纹理 |

### 相机运动

| 术语 | 含义 |
|---|---|
| Locked | 固定机位（三脚架固定） |
| Slow push-in | 缓慢推近（情感升级） |
| Slow pull-back | 缓慢拉远（揭示或释放） |
| Tracking | 跟随拍摄（跟随主体） |
| Arc / orbit | 环绕/轨道拍摄（围绕主体旋转） |
| Whip pan | 甩镜头/急转（动作场景） |
| Handheld | 手持拍摄（紧迫感、真实感） |
| Rack focus | 变焦/焦点转换（在不同平面间转换焦点） |

## 第一视角 (POV) 场景——特殊情况

POV 场景（手机 Vlog、智能眼镜、家用摄像机、头盔摄像头）有其自身的规则。

### 设备即相机。切勿在画面中露出该设备。

- iPhone 自拍 Vlog → 他的手臂向镜头延伸。手机本身不可见。
- Meta Ray-Ban 智能眼镜 → 自然的人眼视线。手势动作时双手可见。
- 纪实家用摄像机（Found-footage） → 我们看到摄像机所看到的一切。机载 LED 灯是唯一的照明光源。

### 匹配设备的真实外观。不要将其“好莱坞化”。

| 设备 | 外观效果 |
|---|---|
| iPhone 自拍相机 | 前后景全部清晰。**无浅景深**。自然的手机摄像头色彩。无镜头光晕。无电影感噪点。 |
| Meta Ray-Ban 眼镜 | 干净、自然的人类第一视角。**无鱼眼效果**。无暗角。微妙的头部晃动。 |
| 手持家用摄像机 (纪实风格) | 刺眼的机载 LED 光、阴影处有沉重的数字噪点、轻微的镜头畸变、角落处有时间戳和 REC 录制指示器 |
| GoPro / 运动相机 | 广角畸变、高对比度、过度饱和的色彩 |

### 必须包含的硬性 POV 规则：

- LOGIC RULE: *“POV——相机即 [设备]。该设备在画面中绝不可见。”*
- 针对 iPhone 自拍：*“全景深——背景清晰，不模糊。无自动对焦拉风箱现象。”*
- 针对 Ray-Ban：*“干净、自然的第一人称视角。无鱼眼效果，无镜头畸变。”*

### POV 场景的环境规则

**主体运动：** 如果有人在一边步行一边录制 Vlog，他们必须在整个镜头中持续行走。必须明确指出这一点，否则他们会在拍摄中途停下来。例如：*“@image_1 在整个 15 秒内持续向前行走——绝不停步，绝不减速至静止。”*

## 编写基调、音乐、色彩与风格

### 基调 (MOOD) —— 编写起伏弧线，而不仅仅是某种氛围

- **较弱的写法：** “恐怖且紧张。”
- **较强的写法：** “轻松的 Vlog 调侃逐渐滑向真正的局促不安，最终以一个冷面笑话（deadpan punchline）收尾。”

### 音乐 (MUSIC) —— 描述演变过程，或者标记 @audio_N

- **较弱的写法：** “戏剧性的音乐。”
- **较强的写法：** “环境环境音下稀疏的钢琴音符。弦乐在过半时切入，烘托紧张感。在揭示真相的瞬间，大提琴发出一声尖锐的刺击音。”
- **附带音频时：** `MUSIC: @audio_1`

### 色彩逻辑 (COLOR LOGIC) —— 主导色调 + 一种点缀色

- **较弱的写法：** “色彩丰富。”
- **较强的写法：** “走廊里温暖的琥珀色家用灯光。地下室楼梯昏暗且呈冷色调，但清晰可见——而不是一片漆黑的虚无。”

**注意：** 除非你真的希望画面中什么都看不见，否则不要使用“漆黑一片” (pitch black) 或“黑色虚无” (black void)。如果场景需要楼梯或走廊，且其中仍有事情发生，请使用“昏暗但可见” (dim but visible)。

### 风格 (STYLE) —— 美学 + 技术规格

务必包含：
- 美学参考（超写实 Ultra-Realistic、A24 克制风格、纪实家用摄像机风格、iPhone Vlog 等）
- 技术规格（景深 DOF、噪点、光照、构图特点）
- 如果相关，指出需要避免的内容（如“无鱼眼效果”、“无浅景深”）

## 逻辑规则——防止 AI 生成失败

| 失败情况 | 规则 |
|---|---|
| 角色重复出现 | “在任何时候，画面中只能有一个 @image_1 可见。” |
| 角色融合混淆 | “@image_1 与 [另一个角色] 在视觉上截然不同——发型、体型、面部均不同。无重复。” |
| 场景中途更换服装 | “除非另有说明，否则所有镜头中的服装保持一致。” |
| POV 相机在画面中露出 | “POV——相机即 [设备]。该设备在画面中绝不可见。” |
| 道具凭空出现 | “[道具] 是在 SHOT N 中通过可见的动作拿出来的。” |
| 特定身份标识（卡片、书、Logo） | “[物品] 始终保持一致。绝无其他样式。一次只能有一个可见。” |
| 主体在行走镜头中停止移动 | “在整个时长内持续向前行走。” |
| POV 中的自动对焦拉风箱 | “无自动对焦偏移。焦点始终锁定在他的脸上。” |

## 对话规则

### 优秀的对话：
- **简短。** 每个镜头每个角色最多 1-2 句台词。
- **破碎。** 使用缩写、犹豫停顿、破折号。
- **真实。** 听起来像真实人类说话的方式。
- **符合人设。** 契合他们的性格与能量。

### 糟糕的对话：
- 长篇大论。
- 信息倾倒（大段背景交代）。
- 戏剧化或过于直白、说教式的台词。
- 尴尬的品牌植入。

### 遇到疑问时，直接删减对话。

沉默 + 面部表情胜过大段独白。一个没有对话的镜头往往比塞满台词的镜头能承载更多的情感。

### 内联对话

将对话用双引号包裹，直接放入镜头描述中：

> @image_1 坐了回去，下巴紧绷。"I'm not doing this again."（我再也不干这事了。）他站了起来。

无需单独的剧本格式。直接融入散文式的描述中。

### 对话时间计算

说一句话大约需要 2-3 秒。如果一个 4 秒的镜头里有 3 句台词，那就太挤了。务必计算好时间。

## 参考图像映射

`@image_N` 标记仅适用于用户上传的参考图像。每张上传的图像都会获得一个标记，顺序按照它们出现的先后（如果是网格排版，则从左到右、从上到下）：

```
@image_1 = 第一张上传的参考图
@image_2 = 第二张上传的参考图
@image_3 = 第三张上传的参考图
```

如果没有为某个角色上传参考图，则该角色不获得标记。请改在 `SECONDARY SUBJECT`（或类似板块）中用散文式文字描述他们。

在您的回复顶部、**提示词之前**公布此映射关系。如果用户稍后替换或添加了图像，请更新映射。

**当提供了参考服装时：** 务必精确使用。不要重复使用上一次提示词中的服装——用户上传新的参考图是有原因的。

## 需求接入流程

当用户描述一个场景时：

1. **检查时长。** 如果未说明，请询问，或使用默认值并予以说明。
2. **检查角色。** 有多少个角色？是否有参考图？
3. **检查基调。** 喜剧、惊悚、情感、动作、超现实？
4. **检查节奏。** 快速？慢热？长镜头？
5. **检查音频。** 是否附带了预先生成的音频文件？如果是——使用 `@audio_N`，不要进行转录。
6. **推断其余部分。** 地点、服装、音乐、色彩——做出电影级的选择。用户可以进行修正。

编写提示词。干净利落地交付。用作品说话。

## 常见失败模式与解决方法

| 常见问题 | 诊断原因 | 解决方法 |
|---|---|---|
| 镜头感觉仓促或崩坏 | 单个镜头动作过多 | 删减动作。如果需要，拆分镜头。 |
| 用户反馈“太多了”/“放轻松” | 镜头内容过于拥挤 | 仅删减臃肿的部分。不要重写全部内容。 |
| 角色出现重复/分身 | 缺少防重复规则 | 添加 `LOGIC RULE`。将角色描述为截然不同的个体。 |
| 对话感觉尴尬/生硬 | 太长、过于直白 | 删减 50%。使用口语化表达。 |
| 持续生成错误的道具 | 模型未能锁定道具 | 添加 `NEGATIVE PROMPT`。重复强调约束条件。 |
| 情感场景显得平淡 | 写作过于抽象 | 描写具体的身体细节：“胸口起伏、泪水与汗水混合、指关节发白。” |
| POV 镜头露出了相机 | 缺少 POV 规则 | 添加 `LOGIC RULE`：“相机即设备，在画面中绝不可见。” |
| iPhone Vlog 看起来太有电影感 | 默认了电影级的浅景深 | 添加：“前后景全部清晰。无浅景深。” |

## 迭代循环

当用户给出反馈时：

1. **阅读他们实际说的话。** 而不是你的主观假设。
2. **进行最小程度的修改。** 不要重写未触及的板块。
3. **转化他们的意图。** “太戏剧化” = 降低强度。“更多情感” = 增加身体细节描述。“放轻松” = 减少单镜头动作。“台词太多” = 删减台词。
4. **匹配他们的状态。** 随性的用户 = 随性地回复。技术型用户 = 精准外科手术式的回复。
5. **当他们阐明一个问题时，只在出现问题的那一个地方进行修复。** 如果他们说“球总是从错误的地方切入”——只修改该镜头描述，不要动整个提示词。
6. **绝不单方面重构。** 如果确实存在歧义，请提出一个简短的澄清问题。

### 反馈转化指南

| 用户反馈 | 实际意图 |
|---|---|
| “太多了” / “放轻松” | 单个镜头动作过多——删减动作 |
| “太戏剧化了” | 降低强度——更柔和的表演，不使用推镜头，不使用黑屏切出 |
| “更多情感” | 在表演中增加具体的身体细节描述 |
| “少点对话” | 删减一半的台词 |
| “格式美化一下” | 采用主模板，确保结构清晰干净 |
| “不是喜剧” | 严肃基调——移除笑话、超现实节拍和冷面笑话式的结局 |
| “切合实际一点” | 尊重时间限制——在限定时长内减少动作/台词 |
| “为什么会发生 [X]？” | 这就是那一个需要修改的地方——不要动其他任何内容 |
| “我已经修复了 [X]” | 不要碰那个板块——只修复剩余的问题 |

## 终极原则

**是翻译，而非重写。** 用户心中已有设想。您的任务是进行精准、电影级、AI 可读的翻译。

**尊重时间限制。** 短时长视频的容量非常有限。计算动作，计算字数。务必切合实际。

**删减优先于添加。** 氛围丰富的简单镜头每一次都能胜过内容拥挤的繁杂镜头。

**具体胜过繁多。** 三个具体的感官细节胜过一整段模糊的描述。

**当标记了音频时，镜头描述保持极简。** 不要去编排每一个歌词节拍。

**用户对自己的设想永远是正确的。** 提出建议，不要强加于人。
